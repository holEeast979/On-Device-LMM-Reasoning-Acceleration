# 🚀 实施方案（按执行顺序）

## 我对 Claude 方案的评价

Claude 给的方案**整体框架正确，可以直接用**，但 Jarvis 的补充（第 7 章）更实际。结合两者，我调整了执行顺序和一些细节。

## 核心原则：先串行跑通，再拆并行

> Jarvis 补充里说的对：**先跑通串行版，拿到 baseline 数据，再拆双进程**。不要一上来就搞解耦。

---

### Phase 1：串行版 MVP（最优先，1-2 天）

**目标**：让 `视频输入 → GP 解析 → I 帧解码 → Resize → 模型推理 → 文本输出` 这条链路跑通。

#### 模块 1：GOP 解析

- 用 `PyAV` 读取视频的 GOP 结构
- 提取每个 GOP 的 I 帧 `packet.size`（码率代理指标）
- 输出：`List[GOPInfo]`，每个包含 `i_frame_pts`、`i_frame_size`

#### 模块 2：帧解码 + 规整化

- 只解码 I 帧（最简策略，先不做打分筛选）
- `cv2.resize` 到 `336×336`
- 转成 `torch.Tensor`，形状 `[1, 3, 336, 336]`

#### 模块 3：GPU 推理

- 加载 Qwen2-VL-7B（你之前实验用过的模型）
- 送入 resize 后的帧，执行推理
- **关键**：ViT 编码后正确清理激活值（`del` → `gc.collect()` → `torch.cuda.empty_cache()`）

#### 模块 4：端到端计时

- 用 `time.perf_counter()` 记录各阶段延迟
- 用 `torch.cuda.memory_stats()` 记录显存 Reserved / Allocated
- 这就是你的 **Baseline 数据**

---

### Phase 2：填充 AV-LRM 稀疏化算法（1-2 天）

在 Phase 1 跑通后，把"全部 I 帧都用"改成"打分筛选"：

#### 模块 5：音频能量提取

- `librosa.load()` 加载音频
- `librosa.feature.rms()` 提取能量曲线
- 按 GOP 时间段切分，得到每个 GOP 的音频能量 $A_i$

#### 模块 6：AV-LRM 打分 + 筛选

- 归一化：$\hat{V}_i$ 和 $\hat{A}_i$ 分别 min-max 归一化到 [0, 1]
- 加权打分：$Score_i = \alpha \cdot \hat{V}_i + (1-\alpha) \cdot \hat{A}_i$
- 方差判断：
    - 方差大 → Top-K 选取（视频有明显高潮低谷）
    - 方差小 → 均匀采样（视频比较平淡）
- **$\alpha$ 先设 0.5，后续消融实验再调**

#### 模块 7：消融实验脚本

- 对比"全量 I 帧" vs "稀疏化后"的延迟和准确率
- 跑一组 $\alpha \in {0.0, 0.3, 0.5, 0.7, 1.0}$ 的消融
- 用 MSVD-QA 或类似 benchmark 评估准确率

---

### Phase 3：CPU-GPU 解耦（1 天）

在串行版确认有效后，拆成双进程：

#### 模块 8：双进程架构

- CPU 进程：解码 → Resize → 写入 `multiprocessing.Queue`
- GPU 进程：从 Queue 读取 → ViT → LLM → 输出
- `Queue.put(None)` 作为结束信号

> ⚠️ **关于 RingBuffer**：Claude 方案里 MVP 用 Queue、后续换 RingBuffer。但 Jarvis 指出两者语义不同（Queue 阻塞等待 vs RingBuffer 覆盖旧帧）。**建议先用 Queue 跑通，如果需要 RingBuffer 再用 `multiprocessing.shared_memory` 实现。**

#### 模块 9：对比实验

- 串行版 vs 双进程版的端到端延迟对比
- 显存使用对比

---

### Phase 4：显存精细优化（1 天）

#### 模块 10：显存清理完善

- 正确的清理姿势（Claude 方案的 `empty_cache()` 不够，要按 Jarvis 说的三步走）：
    
    python
    
    del intermediate_tensors   # 1. 删除引用
    
    import gc; gc.collect()     # 2. 强制垃圾回收
    
    torch.cuda.empty_cache()   # 3. 释放缓存
    

#### 模块 11：KV Cache 策略

- 如果 HuggingFace Transformers 支持 `StaticCache`，直接用
- 否则考虑集成 vLLM 的 PagedAttention
- 记录 KV Cache 的显存曲线

# ⚠️ 我的几点建议（对 Claude 方案的修正）

### 1. 执行顺序要改

Claude 方案的章节顺序是按"技术模块"排列的（Phase 1 搭 MVP → Phase 2 稀疏化 → Phase 3 显存优化），但 **Jarvis 的执行优先级建议更靠谱**：

|顺序|任务|理由|
|---|---|---|
|1|**先跑通串行版**（不解耦、不稀疏化）|验证端到端可行性，排除模型/数据问题|
|2|**加显存监控**|拿到 baseline 数据，不然后面优化没对照|
|3|**再加稀疏化**|这块最容易调参，但也最容易掉准确率|
|4|**最后拆双进程**|解耦带来的收益需要数据支撑|

### 2. `empty_cache()` 的坑要注意

Claude 方案里直接在 ViT 后调 `torch.cuda.empty_cache()`，但这**只释放缓存，不删除张量**。必须先 `del` 再 `gc.collect()` 再 `empty_cache()`。在写代码的时候一开始就写对，别等后面 debug。

### 3. P 帧接口要预留

虽然 MVP 只用 I 帧，但要**预留 P 帧采样的函数签名** `sample_p_frames(gop, strategy)`，方便后续扩展。I 帧间隔通常 1-2 秒，动作类视频只用 I 帧会丢时序信息。

### 4. 自适应 K 值

Claude 方案的 Top-K 是固定的，建议加一个**自适应 K 机制**：视频复杂度高时多采样，低时少采样。这也是一个论文创新点。

### 5. 论文故事线要锐化

Jarvis 说得对，当前方案偏工程 trick 堆叠。建议选一个主打故事深挖：

- **"首个面向端侧的音视频联合稀疏化框架"**（我推荐这个，差异化最强）
- 或 "基于 GOP 感知的多模态推理加速"
