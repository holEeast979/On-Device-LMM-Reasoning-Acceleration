## 1. I 帧与 GOP 格式

### (a) I 帧码率 = I 帧大小？

**是的，在我们的场景中等价。** 严格来说"码率"(bitrate) 是"数据量/时间"，但对于单帧来说，`packet.size`（字节数）就是该帧的数据量。I 帧越大，说明画面细节越多、压缩后保留的信息越多 → 画面越复杂。我们用它作为**画面复杂度的代理指标**。

### (b) GOP 的格式

GOP 是视频编码的概念，不是一种文件格式。H.264/H.265 编码视频时：

```
GOP 0:  [I] [P] [P] [B] [P] [B] [P] ... [P]
GOP 1:  [I] [P] [P] [B] [P] [B] [P] ... [P]
```

- **I 帧**（Intra frame）：完整的一帧画面，不依赖其他帧，可以独立解码
- **P 帧**（Predicted）：只存储与前一帧的差异（运动矢量 + 残差）
- **B 帧**（Bi-predicted）：同时参考前后帧的差异

所以 **I 帧确实就是视频中的一帧画面**，但它是"完整"的那帧。P/B 帧是"差异"帧——你单独拿 P 帧出来是看不到完整画面的，必须从前面的 I 帧开始逐步"还原"。

### (c) I 帧太稀疏，还需要 GOP 筛选吗？

**这是个关键问题。** 让我用数据说明：

以一个 120 秒、30fps 的视频为例：
- 总帧数: 3600
- GOP 数量: ~36（每个 GOP 约 100 帧 / 3.3 秒）
- I 帧数: 36

如果不做 GOP 筛选，直接用全部 36 个 I 帧 → 36 帧送入 ViT Encoder。
如果做 GOP 筛选保留 50% → 18 帧。

**问题在于**：Qwen2.5-Omni 的 ViT Encoder 对每帧产生大量 visual token（一帧 336×336 → 约 256 个 token）。36 帧 = 9216 token，18 帧 = 4608 token。这个差异是很大的——你之前的 token-scaling 实验已经证明 **visual encoder 耗时与 token 数量是线性关系**（R² = 0.986）。

**所以 GOP 筛选是有意义的**，它的价值不在于"减少多少帧"，而在于**减少多少 visual token**。你的两级稀疏化设计是合理的：
- **粗粒度**：GOP 级筛选（选哪些 GOP 值得看）
- **细粒度**：GOP 内部的帧筛选（从选中的 GOP 里选哪些帧）——目前先只取 I 帧，后续可以加 P 帧

## 4. 数据集与编码

### (a) GOP 特征不同的原因

**主要是编码参数的差异，不完全是内容差异。** 同一段视频用不同编码参数可以产生完全不同的 GOP 结构。

### (b) "编码参数"具体指什么

视频编码时有几个关键参数控制 GOP 结构：
- **`-g` (GOP size)**：两个 I 帧之间的最大帧数。比如 `-g 250` 意味着最多 250 帧一个 I 帧
- **`-keyint_min`**：最小 GOP 大小
- **场景检测（scene detection）**：编码器检测到画面剧变时会插入 I 帧，即使没到 `-g` 的上限

Video-MME 的视频 GOP 比较大且稳定（avg 80-156 帧），说明编码时用了较大的 GOP size、场景切换少。ActivityNet 的视频 GOP 变化更剧烈（avg 32-157），说明编码参数更混杂（不同用户上传的视频编码方式不统一），或者视频内容场景切换更频繁。

## 5. 极短 GOP 的处理

**对，你理解得完全正确。** 

极短 GOP（1-10 帧）通常是场景切换瞬间——编码器检测到画面突变，强制插入了一个 I 帧。这种 GOP 的 I 帧码率可能很高（因为与前一帧完全不同），但它只代表了极短暂的画面（0.03-0.3 秒），**信息价值有限**。

所以持续时间加权的逻辑是：**一个 GOP 覆盖的时间越长，说明它代表了更多的视频内容，权重应该更大。** 具体实现可以在 AV-LRM 公式中加一个时间因子，或者简单地设一个最低帧数阈值（比如 GOP 至少 10 帧才参与打分）。这个在 Step 2 里实现。

## 6. GOP Parser 的技术实现

### "Demux 层面解析"是什么意思？

视频处理有两个层次：
1. **Demux（解封装）**：从视频容器（.mp4）中取出压缩的数据包（packet），不解码

## 1. I 帧全解码是什么意思？影响大吗？

**含义**：我们的 [frame_decoder.py](file:///root/scripts/fasteromni/modules/frame_decoder.py:0:0-0:0) 用 `for frame in container.decode(stream)` 遍历视频流——PyAV 按 DTS 顺序**解码每一帧**（包括所有 P/B 帧），我们在解码后检查 `frame.key_frame` 过滤出 I 帧。也就是说：**CPU 侧的视频解码工作量没有减少**，只是最终送给模型的帧少了。

**影响**：**不大**。实测 I 帧解码（含全解码开销）只有 90-160ms，而真正的 TTFT 是 300-2300ms。解码侧不是瓶颈。真正的加速来自**模型侧 visual token 减少**（ViT Encoder + LLM Prefill 的计算量少了）。

**如果要优化**：用 `container.seek(pts)` 跳到 I 帧位置再解码，可以跳过中间帧的解码。但 H.264 的 P/B 帧依赖参考帧，seek 的准确性取决于编码器的 GOP 结构。这个是后续优化项，不影响当前结论。

## Ring Buffer 初步方向和计划

### 核心问题

当前 Pipeline 是**严格串行**的：

```
CPU: [视频解码] → [音频提取] → [预处理] → 等GPU → 等GPU → 等GPU
GPU:              等CPU                    → [ViT] → [Prefill] → [Decode]
```

CPU 和 GPU 互相等待，利用率低。

### Ring Buffer 方案

用一个**环形缓冲区**解耦 CPU 预处理和 GPU 推理：

```
CPU Thread: [解码视频N] → [解码视频N+1] → [解码视频N+2] → ...
                ↓               ↓               ↓
Ring Buffer: [slot0] ────── [slot1] ────── [slot2]
                ↓               ↓               ↓
GPU Thread:  [推理N-1] ←── [推理N] ←───── [推理N+1] → ...
```

**核心设计**：
- **Buffer 大小**：2-3 个 slot（太大浪费显存，太小无法隐藏延迟）
- **每个 slot 内容**：预处理好的 `inputs` dict（已 tokenize + 音频特征，可直接送 model.generate）
- **同步机制**：`threading.Event` 或 `queue.Queue`，CPU 写完一个 slot 通知 GPU
- **关键约束**：slot 中的 tensor 需要 pin_memory（CPU 端），GPU 推理前 `.to(device)` 做异步拷贝

### 实现计划

| 步骤 | 内容 | 依赖 |
|------|------|------|
| 1 | 定义 `RingBufferSlot` dataclass（存 inputs + metadata） | 无 |
| 2 | 实现 `ProducerThread`（CPU 端：视频解码 + 预处理 + tokenize） | Step 1 |
| 3 | 实现 `ConsumerLoop`（GPU 端：从 slot 取数据 → generate） | Step 1 |
| 4 | 实现 `RingBuffer`（管理 slot 的读写指针 + 同步） | Step 1 |
| 5 | 集成到 Pipeline（`run_batch()` 方法） | Step 2-4 |
| 6 | 性能对比（串行 vs Ring Buffer，多视频场景） | Step 5 |

### 预期收益

- **单视频**：几乎没有收益（只有一次推理，没有 overlap 机会）
- **批量视频**：CPU 解码 video N+1 的同时 GPU 推理 video N，**隐藏 preprocess 延迟**
- 预处理约 200-1700ms，如果能完全隐藏，相当于 **端到端延迟减少 10-30%**

### 关键风险

- **显存压力**：每个 slot 需要存一份完整 inputs（含 pixel_values），N 个 slot = N 倍显存
- **需要 pin_memory**：CPU 端 tensor 必须用 pinned memory 才能异步拷贝
- **错误传播**：Producer 线程 OOM/异常需要优雅通知 Consumer

## 显存管理优化初步方向

### 核心问题

ViT Encoder 前向传播产生大量中间激活值，但这些激活值在 Encoder 完成后就不再需要了。当前 PyTorch 默认不主动释放，导致 LLM Prefill 阶段显存紧张。

### 优化方案

**三层递进**：

#### Layer 1：ViT 后显式清理（最简单）

```python
# 在 model.generate() 的 prepare_inputs 阶段 hook 进去
# ViT forward 完成后，显式 del 中间变量 + torch.cuda.empty_cache()
```

实现：注册 forward hook 在 `model.thinker.visual` 上，hook 完成后清理。

#### Layer 2：输入规整化（中等难度）

不同视频的 visual token 数量差异很大（299~12240），导致 GPU 内存碎片化。可以：
- 对齐 token 数量到固定 bucket（如 512/1024/2048/4096）
- 用 `torch.cuda.memory.set_per_process_memory_fraction` 限制峰值

#### Layer 3：分块 ViT Encoding（较复杂）

当帧数很多时，不一次性把所有帧送入 ViT，而是分块处理：
- 每次处理 N 帧（如 8 帧）
- 处理完一块就释放该块的中间激活
- 最终合并所有块的输出 embedding

### 预期收益

- Layer 1：峰值显存下降 10-20%，**允许处理更长视频**（OOM 边界从 ~30s 提升到 ~40s+）
- Layer 2+3：进一步降低显存碎片，可能允许 60s+ 视频

| 步骤 | 内容 | 难度 |
|------|------|------|
| 1 | 用 `torch.cuda.memory_stats()` 测量当前显存 profile | 小 |
| 2 | 实现 ViT forward hook + 清理 | 小 |
| 3 | 对比清理前后的显存峰值 + OOM 边界 | 小 |
| 4 | 实现分块 ViT Encoding | 中 |

我们的代码用 `random.sample(samples, 50)` 采样的是 **QA 对**，不是视频。所以 50 条 QA 分布在 16 个独立视频上。这是一个 bug——应该先采样视频，再取每个视频的 QA。

## 2. 纠正几个误读

### "等间隔采样参数设为 5"
不是参数设为 5。Uniform 策略选 `ceil(valid_gops × keep_ratio)` 个 GOP，是动态计算的。5 只是那些视频恰好有 5 个 valid GOP。

### "TopK 设为 11"
不是 K=11。"Top-K=11" 是说 **11 个视频走了 Top-K 策略**（而不是 Uniform）。每个视频的 K 值 = `ceil(valid_gops × keep_ratio)`。

### "中位数不应该是十几吗"
16 个视频的 GOP 数排序：`[2, 3, 4, 4, 5, 5, 5, 5, 5, 5, 6, 7, 7, 8, 14, 18, 28]`

大部分集中在 4-7（12/16 = 75%），只有 3 个异常值（14, 18, 28）。中位数确实是 5。这是 H.264 编码的典型特征——**短视频（<30s）的 I 帧间隔通常是 2-5 秒**，所以 20s 视频约 4-10 个 GOP。

## 3. 为什么 alpha 不影响结果

即使走 Top-K，**5 个 GOP 选 3 个（kr=0.5）时，不同 alpha 选出的 3 个大概率相同**。举例：

```
视频 5mWZY2XkWSQ 的 5 个 GOP 分数：

alpha=0.0 (纯音频): 排名 [GOP3, GOP1, GOP2, GOP4, GOP5]  → 选前3 = {1,2,3}
alpha=0.5 (均衡):   排名 [GOP2, GOP1, GOP3, GOP4, GOP5]  → 选前3 = {1,2,3}
alpha=1.0 (纯视觉): 排名 [GOP2, GOP1, GOP3, GOP5, GOP4]  → 选前3 = {1,2,3}
```

**不同 alpha 改变了排名顺序，但前 3 名的集合相同**。这就是为什么准确率一样。只有当 GOP 数量多（>15）且选择比例小（kr<0.3）时，alpha 才能让选出的集合产生差异。
